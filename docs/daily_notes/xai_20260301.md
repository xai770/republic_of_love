## Search Page Redesign — Design Discussion

### Problem Statement

The current `/search` page mixes three unrelated hierarchies:
- Sector → Profession (left panel, flat domain bars)
- Qualification Level (right panel, horizontal buttons)
- Location: Bundesland + Berufe side-by-side (right panel intelligence section)

No visual indicator for "all values in scope" vs. "specific selection."
Professions are buried in the intelligence panel, disconnected from their sectors.

### Dynamic Cross-Filtering (Clarified Late in Discussion)

Counts are **not** static national totals. They react to selections in the other panel:
- Selecting **Bayern** on the right → left panel sector/profession counts update to show Bayern-only numbers
- Selecting **IT & Tech** on the left → right panel Bundesland/city counts update to show IT-only numbers
- QL selection also filters both panels

This means precomputed `demand_snapshot` (which already has `location_state × domain_code × berufenet_id`) can support state×domain cross-filtering by summing the right slice. City-level cross-filtering needs the `city_snapshot` to also carry `domain_code`.

### Filter Pills (Top Bar)

Hierarchical pills with parent > child notation:
- `[🏢 IT & Technology > Python Programmer ×]` — sector > profession
- `[❸ Specialist ×]` — QL level
- `[📍 NRW > Köln ×]` — Bundesland > city (tree selection)
- `[📍 Frankfurt 50km ×]` — radius search (map selection, separate from tree)

Pill rows appear only when they have items. Removing a sector pill removes its child profession pills.

### Radius Search vs. Tree Selection

Two coexisting location filter mechanisms:
1. **Tree selection** (right panel) — discrete: pick Bundesländer / cities by name
2. **Radius search** (map) — continuous: geographic circle, boundary-blind

They AND together. When radius is active, tree counts update to show only postings within that radius (the tree becomes a breakdown of the circle).

**Cross-border radius (future-proofing):**
Frankfurt + 50km = Hesse + bits of Rheinland-Pfalz. Trier + 50km = RLP + Luxembourg.
Lörrach + 50km = BaWü + Basel (CH) + Saint-Louis (FR).

When non-German postings arrive, items without a German Bundesland get grouped under their country name at the same tree level as Bundesländer. No special schema needed — `location_country` as fallback when `location_state` is NULL or non-German.

Dynamic counts: selecting a state adjusts sector/profession counts, selecting a sector adjusts location counts. The precomputed `demand_snapshot` (state × domain × profession) supports this via slice-and-sum.
