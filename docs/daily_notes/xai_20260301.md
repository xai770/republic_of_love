## Search Page Redesign — Design Discussion

### Problem Statement

The current `/search` page mixes three unrelated hierarchies:
- Sector → Profession (left panel, flat domain bars)
- Qualification Level (right panel, horizontal buttons)
- Location: Bundesland + Berufe side-by-side (right panel intelligence section)

No visual indicator for "all values in scope" vs. "specific selection."
Professions are buried in the intelligence panel, disconnected from their sectors.

### Dynamic Cross-Filtering (Clarified Late in Discussion)

Counts are **not** static national totals. They react to selections in the other panel:
- Selecting **Bayern** on the right → left panel sector/profession counts update to show Bayern-only numbers
- Selecting **IT & Tech** on the left → right panel Bundesland/city counts update to show IT-only numbers
- QL selection also filters both panels

This means precomputed `demand_snapshot` (which already has `location_state × domain_code × berufenet_id`) can support state×domain cross-filtering by summing the right slice. City-level cross-filtering needs the `city_snapshot` to also carry `domain_code`.

### Filter Pills (Top Bar)

Hierarchical pills with parent > child notation:
- `[🏢 IT & Technology > Python Programmer ×]` — sector > profession
- `[❸ Specialist ×]` — QL level
- `[📍 NRW > Köln ×]` — Bundesland > city (tree selection)
- `[📍 Frankfurt 50km ×]` — radius search (map selection, separate from tree)

Pill rows appear only when they have items. Removing a sector pill removes its child profession pills.

### Radius Search vs. Tree Selection

Two coexisting location filter mechanisms:
1. **Tree selection** (right panel) — discrete: pick Bundesländer / cities by name
2. **Radius search** (map) — continuous: geographic circle, boundary-blind

They AND together. When radius is active, tree counts update to show only postings within that radius (the tree becomes a breakdown of the circle).

**Cross-border radius (future-proofing):**
Frankfurt + 50km = Hesse + bits of Rheinland-Pfalz. Trier + 50km = RLP + Luxembourg.
Lörrach + 50km = BaWü + Basel (CH) + Saint-Louis (FR).

When non-German postings arrive, items without a German Bundesland get grouped under their country name at the same tree level as Bundesländer. No special schema needed — `location_country` as fallback when `location_state` is NULL or non-German.

Dynamic counts: selecting a state adjusts sector/profession counts, selecting a sector adjusts location counts. The precomputed `demand_snapshot` (state × domain × profession) supports this via slice-and-sum.

## Afternoon: Memory Leak Fix (`c655104`)

### Problem
User testing revealed the search page caused Firefox RAM to spike to 100%, making the system unresponsive. Rapid interactions with the map and filter panels triggered runaway memory consumption.

### Root Cause Analysis — 5 Bugs Identified

| # | Bug | Severity | Root Cause |
|---|-----|----------|------------|
| 1 | **Click propagation leak** | Critical | Bundesland GeoJSON click handler called `doSearch()` but didn't stop propagation. Click also hit `map.on('click')`, creating an unwanted geo pin AND calling `doSearch()` a second time. Rapid clicks = unbounded geo markers + doubled API calls. |
| 2 | **Tooltip rebind churn** | High | `layer.bindTooltip()` on every `mouseover` + `layer.unbindTooltip()` on every `mouseout` — created/destroyed tooltip DOM objects at mouse-movement frequency. Firefox GC couldn't keep up. |
| 3 | **No fetch abort** | High | `loadSectorTree()`, `loadLocationTree()`, `loadIntelligence()`, and `loadResults()` fired without AbortControllers. Each `doSearch()` = 5 concurrent requests (~79KB+ each for tree JSON). Rapid clicking = dozens of in-flight responses accumulating in memory. |
| 4 | **Layer cleanup leak** | Medium | `map.removeLayer()` removes from map but doesn't release child markers/event listeners in LayerGroups. `renderCityMarkers()` and `renderBundeslandOverlay()` created new layers each call without explicit cleanup. |
| 5 | **No debounce** | Medium | Each interaction called `doSearch()` immediately. 5 rapid clicks = 5 full search cycles with 25+ API requests. |

### Fixes Applied

1. **`L.DomEvent.stopPropagation(e)`** in Bundesland click handler — map click no longer fires on state selection
2. **Bind tooltip once** in `onEachFeature` setup, use `setTooltipContent()` + `closeTooltip()` instead of bind/unbind cycle
3. **AbortController** on all 5 fetch paths: `previewAbort`, `sectorTreeAbort`, `locationTreeAbort`, `intelligenceAbort`, `resultsAbort` — stale requests cancelled immediately when new search fires; `AbortError` silenced in catch blocks
4. **`clearLayers()`** before `removeLayer()` on all LayerGroups (bundesland, city markers, posting markers, heatmap)
5. **120ms debounce** on `doSearch()` — wraps `_doSearchNow()`, coalesces rapid interactions; queued retry calls `_doSearchNow()` directly to avoid double-debounce

### Verification (Round 1)
- Browser test: rapid-clicked 5 states simultaneously → 0 console errors, all 6 filter pills rendered correctly, no unwanted geo pins
- Cross-filtering still works (9,483 positions found for 6-state selection)
- Tooltip hover works without rebind churn

## Afternoon (Round 2): Canvas Memory Leak — The Real Killer (`5e0d541`)

### Problem
RAM spike persisted after Round 1 fixes. System monitor showed ~20 GB growth (19 GB → 39.2 GB), filling all 39.5 GB RAM + 2.15 GB swap. Something was creating far more memory than stale fetch responses could explain.

### Root Cause
**`L.heatLayer` creates a full-viewport canvas element on every call.** Firefox is notoriously slow to GC detached canvas buffers.

| Metric | Before | After |
|--------|--------|-------|
| Heatmap points per call | 20,713 (uncapped) | 5,000 (capped) |
| Markers per call | 500 | 200 |
| Preview response | 381 KB | 106 KB (-72%) |
| Canvases created per search | 1 new canvas | 0 (reused) |
| GeoJSON layers per search | 1 new L.geoJSON (16 SVG paths) | 0 (style update only) |

**Math**: On a high-DPI display, each canvas ≈ 38 MB. With 120ms debounce, rapid clicking creates ~8 canvases/second. 8 × 38 MB = 304 MB/s of orphaned canvas memory waiting for Firefox GC. Over 60 seconds of interaction → **~18 GB** — matching the observed spike exactly.

### Fixes Applied
1. **Reuse heatLayer** — call `setLatLngs()` + `setOptions()` instead of destroy/recreate. Single canvas lives for the entire session; only the data points are swapped.
2. **Reuse bundeslandLayer** — call `setStyle()` instead of clearLayers/recreate. GeoJSON parsed once at startup; `onEachFeature` handlers bound once; only visual styles update when state selection changes.
3. **Cap heatmap** — server-side limit of 5,000 grid cells (top by weight), down from 20,713 uncapped.
4. **Reduce markers** — 200 instead of 500. Still shows the most recent postings with coordinates.
5. **Increase debounce** — 250ms (was 120ms) to further reduce redraw frequency during rapid interaction.

### Verification (Round 2)
- 20 rapid-fire state toggles (50ms intervals, faster than user can click) → 0 console errors
- All filter pills rendered correctly, map overlay highlights states, heatmap updates in-place
- Preview response: 106 KB (was 381 KB)
